# Specs (스펙 문서)

이 폴더는 V2 이후 기능의 **스펙 문서**를 모아두는 곳입니다.
각 스펙은 두 에이전트의 **공용어** — 기획 에이전트(Cowork Claude)가 작성하고, 구현 에이전트(터미널 Claude Code)가 그대로 따라 만듭니다.

> 두 에이전트 워크플로우의 전체 규칙은 `CLAUDE.md` 의 "작업 분담" 섹션 참조.

---

## 폴더 규칙

- 메이저 버전별로 분리 — `v2/`, `v3/`, ...
- 파일명: `NN-feature-name.md` (예: `01-transport-mode.md`)
- 번호는 **착수 순서** — 의존성을 고려해 부여 (작성 순서 아님)
- 새 스펙을 만들 땐 `_template.md` 를 복사해서 시작

## 스펙의 라이프사이클

각 스펙 파일 최상단에 한 줄로 상태를 표기합니다.

| 상태 | 의미 |
|------|------|
| 📝 작성중 | 기획 에이전트가 초안 작성 중. 아직 검토 전. |
| 🔍 검토중 | 사용자 검토 대기. 코멘트·수정 반영 단계. |
| ✅ 승인됨 | 구현 착수 가능. 구현 에이전트는 이 상태 이상만 작업한다. |
| 🚧 구현중 | 터미널 Claude Code 작업 중. |
| 🎉 구현 완료 | 모든 완료 기준 충족 + 검증 루프 통과. |
| ⛔ 보류 | 우선순위에서 빠짐. 이유를 본문에 기록. |

상태가 `✅ 승인됨` 미만인 스펙은 **구현 에이전트가 절대 손대지 않는다.**

## 작성 규칙 (기획 에이전트용)

- `PRODUCT.md` 의 V2 후보로 등재된 기능만 스펙 문서로 발전시킨다
- 새 V2 후보가 생기면 **`PRODUCT.md` 를 먼저 업데이트** 한 뒤 스펙 작성
- "Out of Scope" 항목을 반드시 명시 — 스코프 크리프 방지가 V2의 핵심
- "완료 기준" 은 **테스트 가능한 문장**으로 작성 (e.g. "투표 화면에서 교통수단 토글 클릭 시 현재 선택값이 시각적으로 강조된다")
- "Open Questions" 가 남아 있는 동안에는 절대 `✅ 승인됨` 으로 올리지 않는다

## 구현 규칙 (구현 에이전트용)

- 작업 시작 전에 해당 스펙 파일을 **끝까지** 읽으세요
- 모호하거나 누락된 부분이 있으면 **멈추고 사용자에게 질문**하세요. 임의 추측 금지.
- "Out of Scope" 의 항목은 절대 건드리지 마세요. 필요해 보여도 스펙 갱신 후에.
- 구현 후 "완료 기준" 체크리스트를 모두 충족하는지 자가 검증.
- 검증 루프 3종 (`pnpm typecheck && pnpm lint && pnpm test`) 통과 후 작업 종료.
- **이 폴더의 문서를 직접 수정하지 마세요.** 변경이 필요하면 사용자에게 보고.

## 회귀 테스트 가이드

V2 기능 구현 시 V1 동작이 깨지지 않는지 확인 필수:

- V1 핵심 플로우 (그룹 생성 → 투표 → 확정 → 결과)가 그대로 동작하는지 e2e 또는 수동 확인
- 변경된 API 라우트의 기존 응답 스키마 호환성 (deprecate 후 단계적 변경)
- DB 마이그레이션이 기존 row에 영향 주지 않는지 (`default` 값 또는 nullable 처리)
